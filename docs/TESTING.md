# TESTING

У цьому проєкті тестування побудоване так, щоб перевіряти і дрібні бізнес-правила, і повні робочі потоки. Unit-тести ловлять помилки в логіці сервісу, а integration-тести підтверджують, що система коректно працює разом із файловим збереженням.

Основний акцент зроблено на сценаріях, які реально важливі для предметної області: заборона подвійного бронювання, перевірка меж місця в залі, коректна обробка недостатньої оплати та відновлення стану після перезапуску.

Тести запускаються стандартно через CTest:

```bash
cmake -S . -B build
cmake --build build --config Debug
ctest --test-dir build -C Debug --output-on-failure
```

Для CI додано окрему перевірку покриття. На Linux-збірці запускається coverage job із quality gate не нижче 70% line coverage. Це простий, але практичний мінімум для навчального проєкту.

Якщо коротко, тестова стратегія вийшла збалансованою: швидкі unit-перевірки для щоденної роботи і кілька інтеграційних тестів для контролю найризикованіших місць.
